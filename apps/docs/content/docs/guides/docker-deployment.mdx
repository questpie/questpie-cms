# Docker Deployment

This guide walks through containerizing a QUESTPIE application for production deployment using Docker. We'll cover the multi-stage Dockerfile, automatic migrations and seeds, worker processes, and a complete `docker-compose.yml` setup.

For a complete working example, see the [TanStack Barbershop example](https://github.com/nicepie/questpie/tree/main/examples/tanstack-barbershop).

## Prerequisites

- Docker 20+ with BuildKit support
- A PostgreSQL database (can run in Docker or as a managed service)
- Your QUESTPIE app building successfully with `bun run build`

## Project Structure

A typical production-ready QUESTPIE project has two runtime processes:

```text
┌─────────────────────────┐     ┌─────────────────────────┐
│   Web Server (web)      │     │   Worker (worker)       │
│                         │     │                         │
│  - SSR / API routes     │     │  - Background jobs      │
│  - Admin panel          │     │  - Email sending        │
│  - Auth endpoints       │     │  - Scheduled tasks      │
│  - Auto migrations      │     │                         │
│  - Auto seeds           │     │                         │
└──────────┬──────────────┘     └──────────┬──────────────┘
           │                               │
           └───────────┬───────────────────┘
                       │
              ┌────────▼────────┐
              │   PostgreSQL    │
              └─────────────────┘
```

## Automatic Migrations & Seeds

QUESTPIE supports running migrations and seeds automatically on app startup. Configure this in your `.build()` call:

```ts title="src/questpie/server/app.ts"
import { migrations } from "../../migrations";
import { seeds } from "../../seeds";

export const app = baseApp.blocks(blocks).build({
  // ... other config
  migrations,
  seeds,
  autoMigrate: true,         // Run pending migrations on startup
  autoSeed: "required",      // Run "required" category seeds on startup
});
```

### Seed Categories

Seeds have three categories that control when they run:

| Category | Purpose | Auto-run in production? |
|----------|---------|------------------------|
| `"required"` | System bootstrap data (site settings, default config) | Yes (via `autoSeed: "required"`) |
| `"dev"` | Demo/development data (sample content, test users) | No — run manually |
| `"test"` | Test fixtures for integration tests | No — run in CI only |

Generate seed files with the CLI:

```bash
# Generate a required seed (runs on every startup)
bun questpie seed:generate --name siteSettings --category required

# Generate a dev seed (run manually for demo data)
bun questpie seed:generate --name demoData --category dev
```

Run seeds manually when needed:

```bash
# Run all pending seeds (all categories)
bun questpie seed

# Run only dev seeds (includes required as dependencies)
bun questpie seed --category dev

# Inside a running container
docker exec <container> bun questpie seed --category dev
```

## Example Dockerfile

This multi-stage Dockerfile creates two build targets: `web` for the SSR server and `worker` for the background job processor.

```dockerfile title="Dockerfile"
# syntax=docker/dockerfile:1.6

# Stage 1: Install ALL dependencies (needed for build)
FROM oven/bun:1.3.0-alpine AS deps
WORKDIR /app
COPY bun.lock package.json ./
RUN bun install --frozen-lockfile

# Stage 2: Build the app
FROM deps AS build
COPY . .
RUN bun run build

# Stage 3: Production-only dependencies
FROM oven/bun:1.3.0-alpine AS prod-deps
WORKDIR /app
COPY bun.lock package.json ./
RUN bun install --frozen-lockfile --production

# Stage 4: Web server
FROM oven/bun:1.3.0-alpine AS web
WORKDIR /app
ENV NODE_ENV=production
ENV PORT=3000
RUN addgroup -S app && adduser -S app -G app
COPY --from=prod-deps --chown=app:app /app/node_modules ./node_modules
COPY --chown=app:app package.json bun.lock ./
COPY --from=build --chown=app:app /app/.output ./.output
COPY --chown=app:app src ./src
COPY --chown=app:app tsconfig.json ./
HEALTHCHECK --interval=30s --timeout=3s --start-period=30s --retries=5 \
  CMD bun -e "fetch('http://127.0.0.1:'+process.env.PORT).then(r=>r.ok?process.exit(0):process.exit(1)).catch(()=>process.exit(1))"
USER app
EXPOSE 3000
CMD ["bun", ".output/server/index.mjs"]

# Stage 5: Background job worker
FROM oven/bun:1.3.0-alpine AS worker
WORKDIR /app
ENV NODE_ENV=production
RUN addgroup -S app && adduser -S app -G app
COPY --from=prod-deps --chown=app:app /app/node_modules ./node_modules
COPY --chown=app:app package.json bun.lock ./
COPY --chown=app:app src ./src
COPY --chown=app:app tsconfig.json ./
USER app
CMD ["bun", "src/worker.ts"]
```

### Key Design Decisions

- **Production dependencies only** — The `web` and `worker` stages use `prod-deps`, not the full `deps` stage. Dev dependencies are only needed during build.
- **Non-root user** — Both stages run as a dedicated `app` user.
- **No entrypoint script** — Migrations and seeds run automatically via `autoMigrate`/`autoSeed` in the app config. No shell script needed.
- **Source files at runtime** — The `src/` directory is needed because QUESTPIE server config (collections, fields, hooks, seeds, migrations) is imported at runtime.
- **Health check** — The web server includes a health check that verifies the HTTP server is responding.

<Callout type="info">
The `CMD` for the web server points to `.output/server/index.mjs` — this is the compiled TanStack Start server entry point. If you use a different framework (Hono, Elysia), adjust the entry point accordingly.
</Callout>

## Build & Run

```bash
# Build both targets
docker build --target web -t myapp-web .
docker build --target worker -t myapp-worker .

# Run
docker run -d \
  -p 3000:3000 \
  -e DATABASE_URL=postgresql://user:pass@db:5432/myapp \
  -e APP_URL=https://myapp.com \
  -e BETTER_AUTH_SECRET=your-secret \
  myapp-web

docker run -d \
  -e DATABASE_URL=postgresql://user:pass@db:5432/myapp \
  -e APP_URL=https://myapp.com \
  -e BETTER_AUTH_SECRET=your-secret \
  myapp-worker
```

## Docker Compose

For local development or simple deployments, use `docker-compose.yml`:

```yaml title="docker-compose.yml"
services:
  postgres:
    image: postgres:17
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: myapp
      POSTGRES_PASSWORD: myapp_password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myapp"]
      interval: 5s
      timeout: 5s
      retries: 5

  web:
    build:
      context: .
      target: web
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://myapp:myapp_password@postgres:5432/myapp
      APP_URL: http://localhost:3000
      PORT: "3000"
      BETTER_AUTH_SECRET: change-me-in-production
      MAIL_ADAPTER: console
    depends_on:
      postgres:
        condition: service_healthy
    restart: unless-stopped

  worker:
    build:
      context: .
      target: worker
    environment:
      DATABASE_URL: postgresql://myapp:myapp_password@postgres:5432/myapp
      APP_URL: http://web:3000
      BETTER_AUTH_SECRET: change-me-in-production
      MAIL_ADAPTER: console
    depends_on:
      postgres:
        condition: service_healthy
    restart: unless-stopped

volumes:
  postgres_data:
```

```bash
# Start everything
docker compose up -d

# Seed with demo data (one-time)
docker compose exec web bun questpie seed --category dev

# View logs
docker compose logs -f web worker
```

## Environment Variables

Essential environment variables for production:

| Variable | Required | Description |
|----------|----------|-------------|
| `DATABASE_URL` | Yes | PostgreSQL connection string |
| `APP_URL` | Yes | Public URL of your application |
| `PORT` | No | HTTP port (default: `3000`) |
| `BETTER_AUTH_SECRET` | Yes | Secret for session encryption |
| `SECRET` | No | General app secret |
| `MAIL_ADAPTER` | No | `console` or `smtp` |
| `SMTP_HOST` | If SMTP | SMTP server host |
| `SMTP_PORT` | If SMTP | SMTP server port |

## Deployment Platforms

### Railway / Render / Fly.io

Most PaaS platforms support multi-stage Dockerfiles. Set the build target via platform config:

- **Railway**: Set `DOCKER_BUILDKIT=1` and use `--target web` in build settings
- **Render**: Use a `render.yaml` with Docker build target
- **Fly.io**: Use `fly launch` with the Dockerfile, configure `[processes]` in `fly.toml` for web and worker

### Kubernetes

For Kubernetes, build both images and create separate Deployments:

```yaml
# Web Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-web
spec:
  replicas: 2
  template:
    spec:
      containers:
        - name: web
          image: myapp-web:latest
          ports:
            - containerPort: 3000
          livenessProbe:
            httpGet:
              path: /
              port: 3000
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: myapp-secrets
                  key: database-url

---
# Worker Deployment (single replica to avoid duplicate job processing)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-worker
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: worker
          image: myapp-worker:latest
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: myapp-secrets
                  key: database-url
```

## Troubleshooting

### Migrations fail on startup

- Ensure `DATABASE_URL` is set and the database is reachable
- Check that the PostgreSQL `pg_trgm` extension is available (required for search)
- Review logs for specific migration errors: `docker compose logs web`

### Worker not processing jobs

- Verify the worker is running: `docker compose ps`
- Ensure `DATABASE_URL` points to the same database as the web server
- pg-boss creates its schema on first run — check for permission issues

### Build fails with `drizzle-kit` errors

The `vite.config.ts` should externalize `drizzle-kit` from the build:

```ts
build: {
  rollupOptions: {
    external: [/^drizzle-kit/],
  },
},
```
