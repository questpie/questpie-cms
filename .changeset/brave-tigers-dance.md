# File-Convention Codegen Architecture

This release replaces the manual `QuestpieBuilder` chain with an **automatic file-convention + codegen architecture**. Each entity lives in its own file, `questpie generate` produces typed entrypoints, and every handler receives a flat, fully-typed `AppContext`.

---

## Breaking Changes

### 1. `QuestpieBuilder` removed

The `q()` factory, `.use()`, and `.build()` chain no longer exist. App composition is handled by codegen.

**Before:**
```ts
import { q } from "questpie";
const builder = q()
  .use(starterModule)
  .use(adminModule)
  .collection("posts", (qb) => qb.fields((f) => ({ title: f.text() })))
  .build();
```

**After:**
```ts
// collections/posts.ts
import { collection } from "#questpie";
export default collection("posts").fields(({ f }) => ({
  title: f.text(),
}));

// modules.ts
import { starterModule } from "questpie";
import { adminModule } from "@questpie/admin";
export default [starterModule, adminModule];

// .generated/index.ts — auto-generated by `questpie generate`
```

### 2. RPC module removed

`rpc()`, `r.fn()`, and `r.router()` are deleted. Use the `functions/` directory with plain object exports instead.

**Before:**
```ts
import { rpc } from "questpie";
const r = rpc();
const searchFn = r.fn({ schema: z.object({ q: z.string() }), handler: async (input) => { ... } });
export const appRpc = r.router({ search: searchFn });
```

**After:**
```ts
// functions/search.ts
import { fn } from "questpie";
export default fn({
  schema: z.object({ q: z.string() }),
  handler: async ({ input, db }) => { ... },
});
```

### 3. Positional callbacks replaced with destructured objects

All builder callbacks now use `({ f }) =>` instead of `(f) =>`.

**Before:**
```ts
collection("posts").fields((f) => ({ title: f.text() }));
```

**After:**
```ts
collection("posts").fields(({ f }) => ({ title: f.text() }));
```

This applies to `.fields()`, `.hooks()`, `.access()`, `.admin()`, `.list()`, `.form()`, `.actions()`, `.sidebar()`, `.dashboard()`, and all other builder callbacks.

### 4. `createFetchHandler` — `rpc` option removed

Functions are automatically available on the app instance. Pass only the app.

**Before:**
```ts
createFetchHandler(app, { rpc: appRpc });
```

**After:**
```ts
createFetchHandler(app);
```

### 5. Module factory functions removed

`admin()`, `audit()`, and `starter()` factory functions are deleted. Modules are now static generated objects imported directly.

**Before:**
```ts
import { starter } from "questpie";
import { admin, audit } from "@questpie/admin";

const builder = q()
  .use(starter())
  .use(admin({ branding: { name: "My App" }, sidebar: (...) => ... }))
  .use(audit({ dashboard: true }));
```

**After:**
```ts
// modules.ts
import { starterModule } from "questpie";
import { adminModule, auditModule } from "@questpie/admin";
export default [starterModule, adminModule, auditModule];

// branding.ts — discovered by admin plugin
export default { name: "My App" };

// sidebar.ts — discovered by admin plugin
export default ({ s, c }) => [ ... ];
```

### 6. Email templates: `render`/`subject` replaced with `handler`

**Before:**
```ts
email({
  name: "welcome",
  schema: z.object({ name: z.string() }),
  render: ({ name }) => <h1>Welcome, {name}!</h1>,
  subject: (ctx) => `Welcome, ${ctx.name}!`,
});
```

**After:**
```ts
email({
  name: "welcome",
  schema: z.object({ name: z.string() }),
  handler: ({ input }) => ({
    subject: `Welcome, ${input.name}!`,
    html: `<h1>Welcome, ${input.name}!</h1>`,
  }),
});
```

- Handler receives full `AppContext` (db, collections, services) — enables async data fetching inside templates
- `sendTemplate({ context: ... })` → `sendTemplate({ input: ... })`
- `@react-email/render` and `@react-email/components` moved to optional peer dependencies
- `InferEmailTemplateContext<T>` deprecated in favor of `InferEmailTemplateInput<T>`

## New Architecture

### File convention

One entity per file, organized by type:

```
src/questpie/server/
├── questpie.config.ts          # Runtime config (db, adapters, secret, plugins)
├── modules.ts                  # Static module imports
├── collections/*.ts            # One collection per file
├── globals/*.ts                # One global per file
├── functions/*.ts              # One function per file (nested folders = nested routes)
├── jobs/*.ts                   # One job per file
├── blocks/*.ts                 # One block per file (admin plugin)
├── routes/*.ts                 # Raw HTTP handlers
├── services/*.ts               # Custom services on context
├── messages/*.ts               # i18n messages (filename = locale)
├── auth.ts                     # Auth configuration
├── locale.ts                   # Content locale config
├── sidebar.ts                  # Sidebar contributions (admin plugin)
├── dashboard.ts                # Dashboard contributions (admin plugin)
├── branding.ts                 # Admin branding (admin plugin)
└── .generated/                 # Auto-generated (gitignored)
    ├── index.ts                # App entrypoint + type augmentations
    └── factories.ts            # Typed collection()/global() with extensions
```

### Codegen

Run `questpie generate` (or `questpie dev` for watch mode) to produce `.generated/index.ts` and `.generated/factories.ts`. The generated code:

- Imports all discovered entities
- Calls `createApp()` with the full definition
- Augments `AppContext` with typed services, collections, globals
- Augments `CollectionBuilder`/`GlobalBuilder` with plugin extensions (`.admin()`, `.list()`, etc.)
- Exports `app`, `appConfig` for use in route handlers and client code

### Context-first: `AppContext`

`AppContext` is an empty interface in core, augmented by codegen with typed services. Every handler context (hooks, access, functions, jobs, routes) extends `AppContext`:

```ts
// In a hook — all typed, zero manual imports:
hooks: {
  afterChange: [async ({ data, db, session, queue, collections }) => {
    await collections.comments.deleteMany({ where: { postId: data.id } });
  }]
}

// In a function:
export default fn({
  schema: z.object({ id: z.string() }),
  handler: async ({ input, db, collections }) => {
    return collections.posts.findOne({ where: { id: input.id } });
  }
});
```

### Proxy-based extensions

No prototype mutation. Plugin extensions (`.admin()`, `.list()`, `.form()`) are applied via Proxy wrappers in generated code with `declare module "questpie"` type augmentations.

### `Registry` interface

Codegen generates a `Registry` interface providing typed autocomplete for relation targets, allowed blocks, etc.:

```ts
f.relation({ to: "posts" })       // ← autocomplete from Registry
f.blocks({ allowedBlocks: ["hero", "cta"] })  // ← autocomplete from Registry
```

### `defineService()` for custom services

```ts
// services/stripe.ts
import { defineService } from "questpie";
export default defineService(() => new Stripe(process.env.STRIPE_KEY));
```

Services are available on `AppContext` in every handler: `({ stripe }) => { ... }`.

### Composable sidebar and dashboard

Modules contribute sidebar sections and dashboard widgets. Contributions are merged automatically by `createApp()`. User-level `sidebar.ts` and `dashboard.ts` are appended last.

## What Didn't Change

- **CollectionBuilder** — `.fields()`, `.hooks()`, `.access()`, `.indexes()` etc. are identical
- **GlobalBuilder** — same API
- **Field system** — `f.text()`, `f.number()`, `f.relation()`, `f.upload()` etc. unchanged
- **Auth** — Better Auth integration unchanged
- **Migrations** — Drizzle-based migration system unchanged
- **HTTP adapters** — `createFetchHandler()` API unchanged (minus removed `rpc` option)
- **Admin UI** — React admin panel, `AdminBuilder` (`qa`), all client-side code unchanged
- **CRUD API** — `collections.posts.find()`, `.findOne()`, `.create()`, `.update()`, `.delete()` unchanged
